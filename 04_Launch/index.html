How It Works — Core Workflow

The problem: Ontario compounding pharmacies must maintain a Risk Assessment Record for every preparation they compound. Each record requires cross-referencing NIOSH lists, SDS documents, WHMIS classifications, OCP standards, and NAPRA complexity guidelines — by hand, for every single recipe. A pharmacy with 120 preparations does this 120 times, then again every 12 months for re-review.

What this platform does: The app opens to a Formulation Library (dashboard) listing all saved formulations. Click + New Formulation to create a recipe, or click an existing card to view/edit. The system looks up each ingredient against the pre-loaded hazard database, evaluates complexity, frequency, and exposure, then generates a complete risk assessment (80% pre-filled). Pharmacist reviews, adjusts if needed, and signs. System sets a 12-month review reminder. 45 minutes → 15 minutes per assessment.

8-Step Rule Engine Pipeline (deterministic — no AI at runtime)

  1. Parse MFR ingredients — extract ingredient names, quantities, active vs. excipient flags
  2. Hazard lookup — for each ingredient, query the local PostgreSQL database; retrieve NIOSH table, WHMIS hazards, GHS category, exposure routes, and recommended PPE
  3. Complexity evaluation — classify preparation as SIMPLE / MODERATE / COMPLEX based on NAPRA criteria: number of steps, precision required, dosage form, equipment needed
  4. Frequency & exposure assessment — evaluate batch size, preparation frequency, and estimated exposure time per session
  5. PPE determination — select glove type (Regular Nitrile / Chemo-rated / Double Chemo), respiratory protection (None / Dust Mask / N95 / PAPR), and other controls based on highest-hazard ingredient
  6. Facility control assignment — determine required engineering controls: general ventilation, local exhaust, Biological Safety Cabinet (BSC), or Containment Ventilated Enclosure (CVE)
  7. Risk level assignment (A / B / C) — apply OCP/NAPRA matrix: NIOSH Table 1 → minimum Level C; NIOSH Table 2 with routine large quantity → Level C; NIOSH Table 2 with occasional small quantity → Level B; non-NIOSH → Level A by default
  8. Rationale generation — compile regulatory references, ingredient citations, and justification text into the structured output

Master Formulation Records — MFR (5 endpoints)

MFRs are the "recipes" — each one captures the product name, pharmaceutical form, compounding method, ingredients with quantities, batch size, storage, and beyond-use date. Each MFR belongs to a pharmacy and was created by a specific user.

MethodEndpointAuthDescription
GET/api/mfrAuthList all MFRs for the authenticated user's pharmacy. Filters: ?status=ACTIVE or DRAFT or ARCHIVED.
GET/api/mfr/:idAuthFull MFR with nested ingredients array (includes ingredient hazard profile). Also includes any linked assessments.
POST/api/mfrAuthCreate a new MFR. Body includes productName, form, route, frequency, batchSize, compoundingMethod, ingredients array (ingredientId + quantity).
PUT/api/mfr/:idAuthUpdate an existing MFR. Updating a NIOSH-related field flags any associated approved assessments as requiring re-review.
PUT/api/mfr/:id/archiveAuthSoft-archive an MFR. Sets status to ARCHIVED. Archived MFRs are excluded from active lists but retained for audit trail.

Risk Assessments (6 endpoints)

The core of the platform. The generate endpoint runs the deterministic rule engine and saves the result. generate-preview runs the same logic without saving — used by the frontend demo and testing. The approve/submit split reflects the two-step regulatory sign-off workflow required by OCP.

MethodEndpointAuthDescription
POST/api/assessments/generate-previewOpenRun rule engine on submitted MFR data without saving to DB. Used by frontend Dev build and integration tests. Returns full assessment JSON including riskLevel, ppeRecommendations, facilityControls, and rationale.
POST/api/assessments/generateAuthGenerate and persist an assessment from an existing MFR. Body: {"mfrId":"..."}. Saves with status DRAFT. Creates first audit log entry.
GET/api/assessmentsAuthList assessments for the pharmacy. Filters: ?status=PENDING_REVIEW, ?status=APPROVED, ?overdue=true (nextReviewDate in the past).
GET/api/assessments/:idAuthFull assessment with linked MFR, ingredient hazards, audit log history, and all generated fields (complexityLevel, exposureRisks, facilityControls, rationale, references).
PUT/api/assessments/:id/submitAuthPharmacist submits assessment for supervisor review. Changes status from DRAFT → PENDING_REVIEW. Logs action in AssessmentAudit.
PUT/api/assessments/:id/approveSupervisorSupervisor approves the assessment. Changes status to APPROVED. Sets approvedById, approvedAt, and nextReviewDate (+12 months). Logs action in AssessmentAudit.

Database Schema (8 Models, 11 Enums)

ORM: Prisma with PostgreSQL. All models use UUID primary keys, snake_case column names, and createdAt/updatedAt timestamps. Foreign key relationships enforce pharmacy-level data isolation (a user from Pharmacy A cannot access Pharmacy B's MFRs or assessments).

Pharmacy

Root entity. Every user, MFR, and assessment belongs to a pharmacy. Stores facility capabilities that affect risk level assignment.

  • facilityLevel — current certified level (A, B, C)
  • hasMechanicalVentilation — affects PPE recommendations
  • hasBSC — Biological Safety Cabinet presence
  • hasEyewash — emergency eyewash station

User

Staff accounts with role-based access. Three roles control what actions are permitted in the API.

  • PHARMACIST — create MFRs, generate and submit assessments
  • SUPERVISOR — everything above + approve assessments
  • ADMIN — everything above + manage ingredient database

Ingredient

The hazard database. Each record is a full safety profile for one compound, compiled from multiple sources.

  • nioshTable — NONE / TABLE_1 / TABLE_2 / TABLE_3
  • whmisHazards — JSON array of hazard codes (e.g., H350, H360)
  • exposureRoutes — JSON: inhalation, skin, ingestion risk levels
  • recommendedPPE — JSON: gloves, respiratory, gown, eye
  • ventilationRequired + ventilationType — engineering controls
  • pubchemCid, healthCanadaDin — source cross-references

MasterFormulationRecord

The recipe. Pharmacy-scoped. Linked to one pharmacy and one creator (User). Can have multiple versions of assessments over time.

  • productName, concentration, form, route
  • compoundingMethod — step-by-step procedure
  • equipmentRequired — JSON array of equipment names
  • beyondUseDate — storage life per USP <795>
  • status — DRAFT / ACTIVE / ARCHIVED

MfrIngredient (Join Table)

Many-to-many link between MFRs and Ingredients. Stores preparation-specific information about each ingredient.

  • quantity — amount in this specific recipe (e.g., "400mg")
  • isActiveIngredient — distinguishes API (active) from excipient
  • Unique constraint on (mfrId, ingredientId) — prevents duplicates

RiskAssessment

The generated compliance document. Stores the rule engine output plus workflow status. Versioned — a new assessment can be generated for the same MFR if ingredients or standards change.

  • riskLevel — A / B / C (OCP facility level required)
  • complexityLevel — SIMPLE / MODERATE / COMPLEX
  • recommendedPPE — JSON: full PPE specification
  • facilityControls — JSON: required engineering controls
  • rationale — human-readable justification text
  • status — DRAFT → PENDING_REVIEW → APPROVED → EXPIRED
  • nextReviewDate — set to approvedAt + 12 months

AssessmentAudit

Immutable log of every status change. Required by OCP for inspections — demonstrates that assessments were properly reviewed and signed. Cannot be deleted.

  • action — GENERATED / SUBMITTED / APPROVED / EXPIRED
  • changedById — who performed the action
  • previousVersion — JSON snapshot of assessment before change
  • notes — optional comment from the reviewer

RegulatoryReference

Versioned local copies of all regulatory source documents. The rule engine cites specific references in generated assessments. Updated offline when standards change.

  • source — NIOSH / OCP / NAPRA / USP / WHMIS
  • title + version + effectiveDate
  • content — JSON: the rules and thresholds used by the rule engine
  • Unique constraint on (source, title, version) — prevents duplicate imports

Offline Data Pipeline

Design principle: All external data is fetched offline (quarterly or on demand) and stored locally in JSON, then seeded into PostgreSQL. No external API calls happen at runtime. This avoids rate limits, guarantees availability, and makes the system work in restricted network environments. The scrapers are in server/src/data/fetchers/.

Tech Stack

Backend

  • Node.js 18+ + Express.js — REST API server, port 3001
  • Prisma ORM — type-safe DB queries, migration management, schema-first design
  • PostgreSQL 14+ — primary datastore; all reference data pre-loaded
  • JWT (jsonwebtoken) + bcryptjs — stateless auth, password hashing
  • pdfjs-dist — PDF parsing for NIOSH list extraction
  • node-html-parser — HTML parsing for Health Canada scraper
  • axios — HTTP client for PubChem and DPD API fetchers

Frontend

  • React 18 + Vite — Dev build UI: Formulation Library dashboard, MFR form, port 5173
  • api.js — typed HTTP client wrapping all 21 backend endpoints
  • aiService.js — orchestrates assessment generation: tries backend first, falls back to local demo logic if backend unavailable
  • ingredientDatabase.js — 30-ingredient offline fallback for demo mode
  • 4 components: MFRForm, IngredientInput (with autocomplete), RiskAssessmentDisplay, App
  • 3 AI providers configurable: Backend (rule engine), OpenAI GPT-4, Claude (via env vars)
